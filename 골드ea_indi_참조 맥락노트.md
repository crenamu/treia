# H1 Breakout System 맥락노트 (시방서)

## 1. 맥락 분석 (14:00 아시아 컷오프 유령 발동 건)

- **결론**: 상호 간섭(충돌)을 막기 위해 매직넘버를 분리하라는 이전의 디버깅 조언이 완벽하게 실현되어 M10과 M3에 살아있던 구형 EA들만 교체 전 설정(14시 종료)으로 동작했고, 신형 EA의 포지션(0.19랏)은 보호된 **성공적인 매직넘버 분리 사례**임.

## 2. 맥락 분석 (17:00 KST 세션 전환 시 부분 청산 현상)

- **현상 (2026-02-26 17:00)**: 모든 포지션(총 5개)이 전체 평단 손실인데 2개만 청산됨.
- **결론**: EA의 논리적 오류가 아니라, **매직넘버 분리에 의한 각 차트별 개별 독립 평가 시스템이 완벽하게 작동한 결과임.** (M10 차트는 그들만의 평균이 수익이었고, M15 차트는 그들만의 평균이 손실이었기에 청산/홀딩 판단이 정확하게 갈림).

## 3. 트레일링 스탑 및 TP(익절) 로직 상세 명세

### A. 트레일링 스탑 (Trailing Stop)

1. **가동 시점**: 포지션 2개 중 `Main A` (1차 익절 타겟)가 TP에 도달해 청산되면, 즉각적으로 `Main B`의 손절가(SL)가 **본절(Break Even)**로 이동하며, 이때부터 발동됨.
2. **스텝(간격)**: 진입 세션의 **Iron Core 전체 높이(TR)**가 기준이 됨. M30 타임프레임 기준일 경우 전체 TR 크기의 **절반(0.5배)**이 1스텝. 가격이 스텝의 2배 수익이 나면 시작점(본절)에서 1스텝 앞서서 쫓아감.

### B. 1차 익절(TP) 비율 로직 해명 (InpTPRatio = 66%)

- **현상**: "지정가(TP)가 진입가와 전고점의 66% 지점이라 캔들 고점 끝에 아슬아슬하게 형성되는 것 같다"는 사용자 피드백.
- **수학적 검증**: 로직 `candidate = p + (swingH - p) * InpTPRatio`는 **사용자의 예상과 100% 일치**함. 즉, 매수(BUY) 시 진입가(`p`)에서 이전 스윙 고점(`swingH`)까지의 남은 거리를 측정한 뒤, 그 거리의 딱 **66%(2/3 지점)**에 TP를 찍고 있음. 66%라는 수치 자체가 절반(50%)을 넘어서 고점(100%)쪽에 매우 가깝게 붙는 비율이므로 시각적으로 고점 근처로 보이는 것이 당연한 수학적 정상 작동임.

## 4. 진입 랏수 (Lot Size) 계산 혁신 개선

- **문제점 (과거)**: 7번(최대 스케일링) 분량의 증거금을 무조건 빼놓아 첫 진입 랏수가 0.01로 쪼그라듦.
- **해결 방안 (Dynamic Lot Sizing)**: 현재 진입가 대비 손절가까지의 실제 물리적 거리를 산출하여, '진짜 들어갈 수 있는 물타기 횟수' 방만큼만 위험 예산을 할당함. 물타기를 거듭하며 손절가와 가까워질수록, **좁아진 거리 대비 남은 예산**을 재산출하여 물타기 랏수를 공격적으로 증폭(Boosting)시킴.

## 5. 수동 청산 시 스케일링 체인 리필(Refill) 현상 원인과 해결책

- **현상 (2026-02-26 21:30)**: "자동 진입한 포지션을 수동으로 청산했는데 EA가 숨도 안 쉬고 곧바로 엄청난 랏수로 재진입(물타기)을 해버림"
- **원인 분석**:
  - 기존 EA의 물타기 로직은 `현재 열려있는 활성 포지션 개수(cnt)`를 기반으로 다음 물타기 목표 간격을 계산했음.
  - 사용자가 두려움에 포지션을 하나 닫아버리면 개수가 하나 줄어들고(예: 3개->2개), EA는 "지금 가격은 3번째 물타기 구간에 와 있는데 포지션은 2개뿐이네? 자리가 비었다!"라고 판단하여 부족한 슬롯을 그 즉시 메우기 위해 시장가 매수를 던짐.
  - 설상가상으로 빈자리를 채우면서, 앞서 `CalcLots`에서 만든 "거리 계산"과 겹쳐서 남은 리스크 한도 어치를 전부 한 방에 박아넣으므로(2.26랏 등) 비정상적인 거대 랏수가 채워지는 원흉이 됨.
- **해결 완료 (g_ScaleCount 분리)**:
  - 활성 포지션 개수에 의존하지 않고, EA 내부적으로 세션 진입 후 물타기가 총 몇 회차까지 **전진했는가**를 독립적으로 기억하는 메모리 체인 변수(`g_ScaleCount`)를 신설함.
  - 이제 사용자가 포지션을 중간에 닫아도, `g_ScaleCount`는 절대 후진하지 않음. 즉, 이미 지나온 자리(빈칸)를 다시 채우는 좀비 진입 현상이 원천 차단되었으며, 무조건 가격이 더 불리한 다음 스텝(새로운 영토)으로 갔을 경우에만 신규 랏을 투입하게 됨.

## 6. 인디케이터 타임프레임별 방향 불일치(동기화 표류) 원인과 해결책

- **현상 (2026-02-27 10:20경)**: "EA는 정확히 한 방향(SELL)으로 진입했는데, 인디케이터를 띄워놓은 1시간봉, 5분봉, 2분봉 차트에서 Asia 박스 크기와 상승/하락 돌파 화살표(마커) 방향이 서로 다르게 표시됨."
- **원인 분석**:
  - 인디케이터가 차트의 `time[i]`를 기반으로 세션의 기준점(`GetSessStartDT`)을 구할 때, 타임프레임(M2, M5 등)의 특성에 따라 야간장/주말 갭 발생 시 `iBarShift` 호출이 엉뚱한 날짜(전날, 혹은 휴일 캔들)의 H1 바를 리턴하는 문제가 있었음.
  - (예: H1 차트에서는 01:00 캔들이 없어서 그 앞전 캔들을 가리키는데, M5 차트에서는 01:00 정각 캔들이 존재한다든지 하는 사소한 오프셋이 모여 엉뚱한 날짜의 고점/저점을 참조해 Iron Core 크기가 다르게 그려짐).
  - 가장 큰 패착은, 스캔 루프 안쪽에서 단순히 H1봉을 가져오는 부분(`int h1Shift = iBarShift(...)`)에 **EA처럼 깐깐한 주말 및 자정 갭 보정 루프(`while(day_of_year != targetDt)`)가 적용되지 않았기 때문**. 결과적으로 M5 차트가 참조하는 H1 Iron Core와 H1 차트가 자기 자신을 참조하는 H1 Iron Core가 달라진 것임.
  - **[추가 원인] 미래 참조(선반영/Peeking) 버그**: 트레이딩뷰는 10시 정각에 BD(돌파 하향점)를 표시하는데, 메타트레이더의 우리 인디케이터는 하위 타임프레임(M2, M5)에서 9시에 미리 화살표를 찍어버리는 문제 발생. 이는 M2 차트가 시간의 흐름(`time[i]`) 속에서 9시 10분 등 9시 정각의 **H1 캔들이 아직 완성(Close)되지 않은 열려있는 시점(Open State)**에도 불구하고 해당 H1 캔들의 *최종 완성 종가*를 끌어와서 돌파로 간주해버리는 치명적인 누수 현상이 있었음.
- **해결 완료**:
  - EA 본체(`H1_Breakout_EA_v2.mq5`)에서 완벽히 검증된 `GetSessH1Shift(sessStart)` 함수를 인디케이터(`H1_Breakout_Indicator_v2.mq5`)에 그대로 복사 이식.
  - 인디케이터에서 `iBarShift`를 호출하던 모든 라인 렌더링, 실시간 돌파 검사, 마커 그리기의 H1 시프트 계산을 오직 `GetSessH1Shift` 단일 함수만 거치도록 강제 통일. 이제 어느 타임프레임 차트를 띄워도 무조건 동일한 "정확한 1시간 타임프레임" 데이터로 Iron Core가 똑같이 통일되어 그려짐.
  - **미래 참조 방지**: 돌파 여부를 스캔하는 루프 내부에 `if(hTime + 3600 > time[i]) continue;` 강제 잠금쇠 제약을 추가. 특정 하위 타임프레임 차트의 시간이 10시 정각을 돌파(달성)하지 못했다면 결코 9시~10시 사이의 H1 캔들 종가를 미리 당겨서 돌파 검사에 사용할 수 없도록 **엄격하게 차단**. 그 결과 트레이딩뷰와 완벽하게 동일한 10시 타이밍에 정확하게 돌파(BD) 아이콘 생성 완료.

## 7. M15 11.18랏(Full Margin) 진입 거절 사태 심층 역순 분석

- **현상 (20:45)**: "EA가 11랏이라는 거대한 진입을 시도하여 [Not Enough Money] 오류를 다수 뿜어낸 뒤 아슬아슬하게 체결됨."
- **원인 분석**:
  1. 당시 리스크 세팅이 **10%** 였음. (1회 손절 시 22,000불 계좌 기준 약 2,200불 손실 발생 세팅)
  2. 아시아 박스 상단(손절 기준 IronH)은 5189.65 였고 진입가는 5188.24 로서, 진입가부터 손절선까지의 거리가 고작 **1.41달러 (초 밀착 상태)** 였음.
  3. EA의 로직은 `리스크(2200불) / 손절거리(1.41불)` 로 랏수를 산출하므로, 짧은 손절 구간 안에 2200불을 태우기 위해 기하급수적으로 큰 **11.18랏**을 요구한 것임.
- **해결 조치**: 시스템적 에러가 아닌 순수한 허용 리스크의 수학적 결과물임. 하지만 사용자 실수(오타)로 10% 이상이 입력될 위험을 방지하기 위해 EA 입력창을 자유 텍스트 창에서 `ENUM_RISK_TIER` (드롭다운 방식: 0.5%, 1%, ..., 10%)로 완전히 개조하여 안전망(Hard Capping)을 씌움.

## 8. 최신 휩쏘(Whipsaw) 장세 역행 및 돌파구 누락 현상 원인과 해결

- **현상**: "트루 리뷰(트레이딩뷰)에서는 정각 11시에 상방 돌파(BU)를 다시 그렸는데 메타 인디케이터는 반영하지 못함."
- **원인 분석**:
  1. (시간 오차): 서버 시간 산출시 단 1초의 틱 오차로 `04:00 + 3600 > 04:00` 이 스킵되어 정각 캔들을 놓치는 현상. (`3599`초로 인하하여 1초 버퍼 제공).
  2. (로직 맹점): 한번 돌파가 일어난 뒤 반대편으로 다시 뚫리는 **휩쏘(상하 진동)** 장세에서, 과거 코드는 루프 내부에 `break;` 문을 두어 **최초의 돌파 기록**을 만나면 영영 스캔을 멈춰버렸음.
- **해결 완료**:
  - 스캔 로직에서 `break`문을 무조건 `continue` 로 치환 및 삭제하여, 가장 과거의 H1부터 제일 최신의 H1까지 **모든 마감 캔들을 전수 검사**하도록 개조.
  - 최신 방향으로 `brkDir`과 시간(`latestBrkTime`)을 엎어치기(Overwrite) 한 뒤, 해당 시간에 맞는 마커 렌더링. 또한 하방 돌파 화살표는 가시성 강화를 위해 `clrOrange`(주황색)로 변경함. EA 또한 최신 돌파 방향으로 유동적으로 포지션 트리거 방향을 스위치함.

## 9. 수학적 비율 트레일링의 한계와 물리적 데이터 교정

- **현상**: M15는 25%, M5는 8.3% 식의 단순 비율은 시장의 실제 변동성 분포와 맞지 않음. (실제 5분봉은 1시간 봉의 절반 크기일 수도 있음).
- **원인 분석**: 타임프레임이 작아질수록 변동성은 비선형적으로 움직이므로, "가정"에 기반한 코딩은 자칫 너무 먼 익절이나 너무 짧은 익절을 유발함.
- **해결 완료 (2026-03-01)**:
  - EA 내부 전역 변수에 `g_M5TR`, `g_M15TR` 등 실제 데이터 저장소 신설.
  - 세션 전환 시점에 각 타임프레임별 `iHigh - iLow`를 직접 계산하여 메모리에 락(Lock)을 걸어둠.
  - 이제 트레일링 스탑은 "상상 속의 5분 크기"가 아닌, **"오늘 아침 실제 터진 5분 봉의 크기"**를 1단위로 보고 정확하게 가격을 따라감.
  - `SaveState/RestoreState`를 통해 MT5 재시작 시에도 해당 세션의 물리적 데이터가 증발하지 않도록 보강함.
